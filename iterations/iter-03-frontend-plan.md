# Iteration 03 — 前端双应用实施计划

> 目标：在当前 Go 微服务后端基础上，新增 `frontend-user` 与 `frontend-admin` 两个前端应用，先完成可演示、可联调、可持续迭代的第一版
> 背景：项目以学习和作品展示为主，优先保证业务闭环、工程结构清晰、便于后续扩展
> 依赖：现有 API Gateway `:8080`、Swagger/OpenAPI 文档、`api/` 下的服务契约

状态说明：⬜ 待开始 / 🔄 进行中 / ✅ 已完成

---

## 目标范围

本次前端建设不追求一次性做全，而是分成两个独立应用：

- `frontend-user`：面向普通用户的商城前台
- `frontend-admin`：面向运营/管理的后台控制台

目标是先把核心业务链路跑通：

- 用户端：注册/登录 → 浏览商品 → 搜索/筛选 → 加购物车 → 下单 → 支付流程占位 → 查看订单
- 管理端：登录态接入 → 商品管理 → SKU 管理 → 分类管理 → 用户管理 → 订单管理 → 秒杀活动管理

---

## 当前后端能力判断

根据当前 `README`、`configs/dev/gateway.yaml`、`api/*.proto` 的定义，后端已具备较完整的前端接入基础：

- 用户：注册、登录、用户信息、地址管理
- 商品：商品、SKU、分类、Banner
- 购物车：增删改查、勾选结算
- 订单：创建、查询、取消、确认收货
- 支付：创建支付单、查询支付状态、退款
- 搜索：商品搜索、搜索建议、热词
- 营销：优惠券、促销、积分
- 秒杀：活动列表、详情、下单、后台管理
- 评价：创建评价、评价列表、统计、回复

需要明确的限制：

- 当前未看到完整的管理员独立认证、角色、权限、RBAC 设计
- 因此 `frontend-admin` 第一版定位为“管理控制台 UI”，权限体系先做前端预留，不假设后端已具备完整后台用户体系

## 接口对接策略

### 请求层统一约定

- 所有请求统一走 API Gateway `:8080`
- Token 统一存储在本地，并由请求拦截器自动注入
- 对于返回结构统一适配：
  - `code === 0` 或后端约定的成功码视为成功
  - 其余情况统一进入错误处理

### 类型策略

- 第一版先手写前端业务类型，保持开发速度
- 后续如接口趋于稳定，再考虑从 Swagger 或 proto 自动生成类型

### 容错策略

- 对支付、统计、后台权限这类暂未完全稳定的能力，前端先提供占位和兜底逻辑
- 对缺失接口不做假实现，明确标注“预留能力”

## 风险与限制

| 风险 | 说明 | 处理方式 |
|---|---|---|
| 管理员鉴权不完整 | 暂未看到独立后台用户/RBAC 契约 | 后台先做前端守卫骨架，后续再补后端 |
| 仪表盘统计接口不足 | 当前更偏业务接口，不是后台聚合接口 | 第一版先做占位统计或前端组合展示 |
| 支付流程偏模拟 | 更适合演示，不适合真实支付接入 | 前端做“支付流程展示”而非真实支付集成 |
| 接口字段可能继续变化 | 目前后端仍在迭代中 | 请求层集中封装，减少页面直接耦合 |

---

## 执行顺序建议

建议按下面顺序推进，而不是两个前端同时大面积开工：

1. 先搭好两个前端项目脚手架
2. 优先完成 `frontend-user` 的 P0 页面
3. 跑通登录、商品、购物车、下单、订单闭环
4. 再开始做 `frontend-admin` 的 P0 模块
5. 最后补 P1 能力和体验优化

这样做的原因：

- 用户端最能验证后端主链路是否真的可用
- 管理端更依赖接口边界稳定，放在第二波更稳
- 学习型项目更适合先做“看得见的主流程”
